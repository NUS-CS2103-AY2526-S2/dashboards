**1. :fas-info-circle: Java coding standard: How verbose is too verbose?**
</div>

Hi, I read from the book that we should ["make the code as explicit as possible, even if the language syntax allows them to be implicit"](https://nus-cs2113-ay2526s1.github.io/website/se-book-adapted/chapters/codeQuality.html#make-the-code-obvious). Since Java can allow one to be quite verbose in its code, I have the following questions:


# 1. Should I qualify all usage of static members with class name?

For the following Java code snippet:

'''java
final class Duke ``{``
    private static final int MAX_TASKS = 100;
    private static final Task[] tasks = new Task[MAX_TASKS];
    //                                           ~~~~~~~~~
``}``
'''

IntelliJ IDEA gives the following warning:

'''
Unqualified static field access 'MAX_TASKS'

Inspection info: Reports usage of static members that is not qualified with the class name.

This is legal if the static member is in the same class, but may be confusing.
'''

Does the coding standard recommend removing ambiguity by qualifying all usage of static members with the class name? If so, will it be strictly enforced?

'''diff
-    private static final Task[] tasks = new Task[MAX_TASKS];
+    private static final Task[] tasks = new Task[Duke.MAX_TASKS];
'''


# 2. Should I qualify all usage of fields with `this`?

For the following Java code snippet:

'''java
final class Task ``{``
    // ...
    char getStatusIcon() ``{``
        return isDone ? 'X' : ' ';
        //     ~~~~~~
    ``}``
``}``
'''

IntelliJ IDEA gives the following warning:

'''
Instance field access 'isDone' is not qualified with 'this'

Inspection info: Reports field access operations that are not qualified with this or some other qualifier.

Some coding styles mandate that all field access operations are qualified to prevent confusion with local variable or parameter access
'''

Does the coding standard recommend removing ambiguity by qualifying all usage of fields with `this`? If so, will it be strictly enforced?

'''diff
-        return isDone ? 'X' : ' ';
+        return this.isDone ? 'X' : ' ';
'''


# 3. Should all casts be explicitly written?

For the following Java code snippet from [one of Week 4's key exercises](https://nus-cs2113-ay2526s1.github.io/website/schedule/week4/topics.html#key-exercise-print-shape-area):

'''java
public class Circle extends Shape ``{``
    private int radius;
    // ...
    @Override
    public int area() ``{``
        return (int)(Math.PI * radius * radius);
        //     ~~~~~           ~~~~~~   ~~~~~~
    ``}``
``}``
'''

I noticed that while the narrowing cast from `double` to `int` is explicitly casted, the widening cast from `int` (`radius`) to `double` is not explicitly casted.

What are the requirements for needing to explicitly write the cast v.s. it being okay to not write them?

## What to check (functional)

1. **Run from an empty folder**

   * Open a terminal in an **empty** folder.
   * Copy the JAR there and run:

     ```
     java -jar "nami.jar"
     ```
   * Expect the greeting:

     ```
     Hello! I'm Nami
     What can I do for you?
     ```

2. **Add tasks**

   * Todo: `todo read book`
   * Deadline (ISO date input): `deadline return book /by 2019-10-15`

     * Expect pretty date in output: `(by: Oct 15 2019)`
   * Event (free-text times): `event project meeting /from Mon 2pm /to 4pm`

3. **List**

   * `list`
   * Expect 1-based numbering and status boxes, e.g. `[T][ ]`, `[D][ ]`, `[E][ ]`.

4. **Mark / Unmark**

   * `mark 2`: shows `[X]`
   * `unmark 2`: back to `[ ]`

5. **Delete**

   * `delete 3`: removes the event and shows updated count.

6. **Find (case-insensitive substring on description)**

   * `find book`: shows any tasks whose **description** contains ‚Äúbook‚Äù.

7. **Auto-save & persistence**

   * Check that `./data/nami.txt` is created next to the JAR.
   * Type `bye` to exit. Re-run the JAR in the **same folder**.
   * `list` should show the tasks you added earlier.
   * (Optional) Move the JAR to a **different empty folder** and run again: it should start with an empty list and create a new `./data/nami.txt` there.

---

## What to check (error handling)

Please try these and confirm you get a friendly error message:

* Empty input: just press Enter.
* Extra args where not allowed:

  * `list now`: error like `'list' does not take any arguments.`
* Todo without description:

  * `todo`
* Deadline issues:

  * Missing `/by`: `deadline return book`
  * Missing date: `deadline return book /by`
  * Wrong date format: `deadline return book /by 15-10-2019` (should ask for `yyyy-MM-dd`)
  * Missing description: `deadline /by 2019-10-15`
* Event issues:

  * Missing flags: `event meeting`
  * Missing `/from` or `/to` values: `event x /from Mon` or `event x /to 4pm`
* Index problems (list size permitting):

  * No index: `mark`
  * Non-number: `mark abc`
  * Out of range: `mark 999`
* Unknown command:

  * `blah`

* Finally, exit with:

  * `bye`

Hey, I'll answer no. 2.

I think IntelliJ‚Äôs warning is just pointing out one possible convention but it‚Äôs not really a universal standard. I‚Äôm not seeing this warning in my IDE for the same code snippet, so I think it might depend on your inspection settings in IntelliJ.

According to the optional [Java intermediate and advanced rules](https://se-education.org/guides/conventions/java/index.html), we are recommended to avoid unnecessary use of `this` with fields. The `this` keyword is only needed when a field is shadowed by a method of constructor parameter. The rationale for this is to reduce unnecessary noise.

```
// 'name' and 'age' parameters shadow the fields so its necessary
public User(String name, int age) ``{``
    this.name = name;
    this.age = age;
``}``
```


```
// unnecessary 'this.' everywhere
public void move(int dx, int dy) ``{``
    this.x = this.x + dx;
    this.y = this.y + dy;
``}``

// Good version
public void move(int dx, int dy) ``{``
    x = x + dx;
    y = y + dy;
``}``
```
I know it's an exaggerated example but you can see how it can get more unreadable or even tiring to see 'this' repeated a lot when there's no need to distinguish the field x and y from the parameters.


So in this case:
```
return isDone ? 'X' : ' ';
```
is perfectly fine and preferred over
```
return this.isDone ? 'X' : ' ';
```

TLDR
Use `this` only when necessary to distinguish a field from a parameter or local variable. Otherwise it‚Äôs optional and can be omitted for cleaner code.
